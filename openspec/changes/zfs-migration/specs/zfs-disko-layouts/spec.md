## ADDED Requirements

### Requirement: Single-disk ZFS-on-LUKS disko layout
Each single-disk machine (britton-fw, bonsai, aspen1, aspen2) SHALL have a disko.nix that creates a GPT partition table with an ESP partition (unencrypted), an optional swap partition (unencrypted), and a LUKS2-encrypted partition containing a ZFS pool named `rpool` with datasets for root, nix, home, and var-log.

#### Scenario: Single-disk laptop with swap and LUKS
- **WHEN** disko formats britton-fw or bonsai
- **THEN** the disk has: 1GB ESP (vfat, /boot), 8GB swap partition, and remaining space as a LUKS2 partition containing ZFS pool `rpool`

#### Scenario: Single-disk server with LUKS, no swap
- **WHEN** disko formats aspen1 or aspen2
- **THEN** the disk has: 500MB ESP (vfat, /boot) and remaining space as a LUKS2 partition containing ZFS pool `rpool` (no swap partition â€” servers use zram only)

### Requirement: Multi-disk ZFS-on-LUKS disko layout
The britton-desktop machine SHALL have a disko.nix that creates two separate LUKS2-encrypted ZFS pools: `rpool` on the 2TB NVMe (Samsung 9100 PRO) and `datapool` on the 4TB NVMe (Samsung 9100 PRO). Each drive's ZFS partition SHALL be independently LUKS-encrypted.

#### Scenario: Desktop dual-pool formatting with LUKS
- **WHEN** disko formats britton-desktop
- **THEN** the 2TB drive has: 1GB ESP, 8GB swap, remaining as LUKS2 â†’ `rpool`; the 4TB drive has: 100% as LUKS2 â†’ `datapool` with a single `data` dataset mounted at `/data`

### Requirement: LUKS2 encryption parameters
All LUKS partitions SHALL use LUKS2 format with `aes-xts-plain64` cipher, 512-bit key size, and `argon2id` key derivation.

#### Scenario: LUKS device inspection
- **WHEN** `cryptsetup luksDump /dev/disk/by-partlabel/<partition>` is run
- **THEN** output shows Version 2, cipher `aes-xts-plain64`, key size 512 bits, and PBKDF `argon2id`

### Requirement: LUKS device naming in disko
Each LUKS container SHALL have a predictable device-mapper name derived from the machine and pool name (e.g., `cryptroot` for rpool, `cryptdata` for datapool).

#### Scenario: LUKS device mapper names
- **WHEN** LUKS containers are opened during boot
- **THEN** `/dev/mapper/cryptroot` exists for the rpool LUKS device, and `/dev/mapper/cryptdata` exists for the datapool LUKS device (desktop only)

### Requirement: ZFS dataset hierarchy
Every `rpool` SHALL contain datasets: `root` (mounted at `/`), `nix` (mounted at `/nix`), `home` (mounted at `/home`), `var-log` (mounted at `/var/log`), and `reserved` (no mountpoint, 5% reservation for CoW headroom).

#### Scenario: Dataset creation and mount
- **WHEN** a machine boots after disko formatting and LUKS unlock
- **THEN** `zfs list` shows all 5 datasets under rpool, and `df` shows `/`, `/nix`, `/home`, `/var/log` mounted from their respective datasets

#### Scenario: Reserved space prevents pool full
- **WHEN** rpool usage approaches capacity
- **THEN** the `reserved` dataset's reservation ensures at least 5% of pool capacity remains free for CoW metadata operations

### Requirement: Legacy mountpoints
All ZFS datasets SHALL use `mountpoint=legacy` and be mounted via `/etc/fstab` entries generated by disko, not ZFS native mount properties.

#### Scenario: Mount ordering
- **WHEN** the system boots and LUKS is unlocked
- **THEN** ZFS datasets are mounted by systemd fstab units in the correct dependency order (root before nix, home, var-log), after `systemd-cryptsetup` opens the LUKS container

### Requirement: Compression on all datasets
All ZFS datasets SHALL have `compression=zstd` set as a pool-level default.

#### Scenario: Nix store compression
- **WHEN** nix builds store paths into `/nix`
- **THEN** `zfs get compressratio rpool/nix` shows a compression ratio greater than 1.0x

### Requirement: Device identification by stable path
Every disko disk device SHALL be specified by `/dev/disk/by-id/` path, not by `/dev/sdX` or `/dev/nvmeXnY`.

#### Scenario: Disk identity stability
- **WHEN** the kernel enumerates disks in a different order after reboot
- **THEN** disko, LUKS, and ZFS still reference the correct physical device because the by-id path is stable

### Requirement: GRUB boot from ZFS-on-LUKS
All machines SHALL continue using GRUB as the bootloader with ZFS boot support enabled. GRUB SHALL be configured to unlock the LUKS container during early boot via `boot.initrd.luks.devices`.

#### Scenario: Boot after ZFS-on-LUKS install
- **WHEN** the machine boots after nixos-install with LUKS+ZFS root
- **THEN** GRUB loads the kernel and initrd, systemd-cryptsetup prompts for the LUKS passphrase (console or SSH), the LUKS container opens, ZFS imports rpool, and the system boots

### Requirement: ESP remains unencrypted
The ESP (EFI System Partition) SHALL NOT be encrypted. It contains GRUB, the kernel, and initrd â€” all of which must be readable before LUKS unlock.

#### Scenario: ESP accessibility without passphrase
- **WHEN** the machine powers on
- **THEN** GRUB loads from the unencrypted ESP without requiring any passphrase
