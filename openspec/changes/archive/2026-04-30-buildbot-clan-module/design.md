## Context

Buildbot-nix provides two NixOS modules: `buildbot-master` and `buildbot-worker`. The current setup imports both directly into `machines/aspen1/buildbot.nix` with hand-wired clan vars generators for secrets. The master manages a workers JSON file, a GitHub App key, OAuth secret, and webhook secret. The worker reads a password file. Both share a password generated by the same vars generator.

Every other service in this repo follows the clan perInstance pattern: a module in `modules/`, an inventory instance in `inventory/services/`, and tag/machine-based role assignment. Buildbot is the only exception.

The harmonia binary cache runs on aspen1 alongside buildbot — builds land in the local store and are immediately served via harmonia. This co-location is intentional and should be preserved.

## Goals / Non-Goals

**Goals:**
- Move all buildbot config into a clan service module with master and worker roles
- Preserve exact functional parity with the current `machines/aspen1/buildbot.nix`
- Keep clan vars generator names identical so existing SOPS-encrypted secrets don't need re-generation
- Make it possible to add workers on other machines by adding machine references to the inventory instance
- Follow the same patterns as existing modules (harmonia, prometheus, tailscale)

**Non-Goals:**
- Adding new workers right now — just make it possible
- Changing buildbot-nix configuration (domain, GitHub app, eval workers, etc.)
- Implementing scheduled effects (separate change)
- Remote workers over Tailscale (requires master URL configuration changes — future work)
- Multi-master or HA setups

## Decisions

### 1. Role structure: `master` and `worker`

**Choice:** Two roles — `master` (single-machine) and `worker` (any machine).

**Rationale:** Maps directly to buildbot-nix's two NixOS modules. The master role handles `services.buildbot-nix.master` config, GitHub auth, the workers JSON, and nginx/firewall. The worker role handles `services.buildbot-nix.worker` config and password auth. On aspen1, both roles are assigned to the same machine (co-located master + worker), matching the current setup.

**Alternative considered:** Single `server` role that bundles both — rejected because the whole point is to separate master from worker for future distribution.

### 2. Secret sharing: master generates, workers consume

**Choice:** The master's perInstance generates the worker password via clan vars (`buildbot-worker` generator). Workers reference the same generator output. For co-located master+worker on the same machine this works directly. For remote workers, the password would need to be shared via clan vars `share = true`.

**Rationale:** The current setup generates password + workers JSON in a single generator. This stays the same. The workers JSON lists worker names and passwords — the master needs it. Each worker only needs its own password file.

**Trade-off:** Remote workers can't use the same generator output path directly. When remote workers are added later, the generator will need `share = true` or a separate per-worker generator. For now, co-located only, so this is a non-issue.

### 3. Vars generator names: keep them identical

**Choice:** Generator names stay `buildbot-worker` and `buildbot-github`, exactly as they are today.

**Rationale:** Changing generator names would invalidate the existing SOPS-encrypted vars under `vars/per-machine/aspen1/`. Keeping them identical means zero secret rotation and zero re-encryption.

### 4. Settings interface: freeformType with high-level wrappers

**Choice:** Both roles use `freeformType = attrsOf anything` with named options for the most common settings (`domain`, `buildSystems`, `admins`, `evalWorkerCount`, `evalMaxMemorySize`, `outputsPath`, `postBuildSteps` for master; `workers` for worker). The freeform passthrough feeds remaining attrs into `services.buildbot-nix.master` / `services.buildbot-nix.worker`.

**Rationale:** Same pattern as prometheus and harmonia modules. High-level options document the most-used knobs. Freeform passthrough ensures any buildbot-nix option is accessible without module changes.

### 5. GitHub auth config: in inventory settings, not in the module

**Choice:** GitHub App ID, OAuth ID are set in the inventory service instance settings. Secret file paths are injected by the module from clan vars generators (not user-configurable).

**Rationale:** App IDs are non-secret, deployment-specific values that belong in inventory. Secret file paths are always generated by clan vars — there's no reason to let the user override them.

### 6. buildbot-nix module imports: inside the perInstance nixosModule

**Choice:** The perInstance nixosModule does `imports = [ inputs.buildbot-nix.nixosModules.buildbot-master ]` (or `buildbot-worker`). The `inputs` are threaded through as a module argument.

**Rationale:** This keeps the buildbot-nix dependency contained inside the clan module rather than requiring every machine to import it. Only machines that actually get the buildbot roles will import the buildbot-nix NixOS modules.

**Constraint:** The clan service module function receives `{ inputs, ... }` when called from `modules/default.nix`. The harmonia and other modules don't need external inputs, but buildbot does. The module definition is a function `{ inputs, ... }: { ... }` — the outer inventory layer passes `inputs` when importing.

### 7. postBuildSteps: in inventory settings

**Choice:** The `postBuildSteps` (ntfy notification script) is configured in the inventory service instance, not baked into the module.

**Rationale:** Notification targets and scripts are deployment-specific. The module should be generic. The inventory instance can define whatever postBuildSteps it needs.

## Risks / Trade-offs

- **[Vars path coupling]** The module hardcodes generator names (`buildbot-worker`, `buildbot-github`) to match existing SOPS vars. If these names need to change later, it requires secret re-generation. → Acceptable — name stability is a feature here.
- **[Single-machine workers only]** The initial module only supports co-located workers (same machine as master). Remote workers need additional work (shared secrets, master URL config). → Documented as future work, doesn't block current use.
- **[buildbot-nix input threading]** The module needs `inputs.buildbot-nix` which isn't available through the standard clan module interface. → Pass `inputs` from `modules/default.nix` into the module definition function.
- **[Freeform config collisions]** If a user sets a freeform attr that conflicts with a module-managed attr (e.g., `workersFile`), the module's `mkDefault` will lose. → Module uses `mkForce` for attrs that must be module-managed (secret file paths), `mkDefault` for everything else.
