## ADDED Requirements

### Requirement: Theme sync script reads Noctalia color state
The `noctalia-theme-sync` script SHALL read Noctalia's current Material You color palette and write override config files for all themed apps. The script SHALL be generated by Nix at build time (so it knows which apps are active) and installed to the user's PATH.

#### Scenario: Script reads current colors
- **WHEN** `noctalia-theme-sync` runs
- **THEN** it reads Noctalia's current color state (Material You primary, secondary, surface, error, etc.) and maps them to the unified theme color slots

#### Scenario: Script generates all override files
- **WHEN** `noctalia-theme-sync` completes
- **THEN** override files exist for every themed app (fish, starship, helix, btop, eza, delta, bat, swayosd)

#### Scenario: Script is idempotent
- **WHEN** `noctalia-theme-sync` runs twice with unchanged Noctalia colors
- **THEN** all output files are identical and no app restarts are triggered

### Requirement: Noctalia hooks trigger theme sync
The Noctalia `hooks.wallpaperChange` and `hooks.darkModeChange` settings SHALL call `noctalia-theme-sync` so all apps update when the color palette changes.

#### Scenario: Wallpaper change propagates colors
- **WHEN** the user changes wallpaper in Noctalia
- **THEN** the `wallpaperChange` hook calls `noctalia-theme-sync` and all apps receive updated colors

#### Scenario: Dark mode toggle propagates colors
- **WHEN** darkman toggles dark/light mode
- **THEN** the `darkModeChange` hook calls `noctalia-theme-sync` and apps switch to the appropriate variant

### Requirement: Fish runtime color override
The fish shell config SHALL source `~/.config/fish/conf.d/noctalia-colors.fish` which the sync script generates. The file SHALL contain `set -g fish_color_*` commands matching the current Noctalia palette.

#### Scenario: New fish shell uses Noctalia colors
- **WHEN** a new fish shell opens after `noctalia-theme-sync` has run
- **THEN** the shell uses the Noctalia-generated colors from `conf.d/noctalia-colors.fish`

### Requirement: Helix runtime theme override
The sync script SHALL write helix theme files (`noctalia-dark.toml`, `noctalia-light.toml`) to `~/.config/helix/themes/`. Helix auto-watches theme files and SHALL apply changes without restart.

#### Scenario: Helix applies new colors without restart
- **WHEN** the sync script writes `noctalia-dark.toml`
- **THEN** an open helix instance detects the change and applies the new theme

### Requirement: btop runtime theme override
The sync script SHALL overwrite `~/.config/btop/themes/custom-theme.theme`. btop watches theme files and SHALL apply changes on its next refresh.

#### Scenario: btop applies new colors
- **WHEN** the sync script writes the btop theme file
- **THEN** btop applies the new colors on its next screen refresh

### Requirement: Starship runtime palette override
The sync script SHALL update the palette section in starship's config. Starship re-reads its config on each prompt render.

#### Scenario: Next prompt uses new starship colors
- **WHEN** the sync script updates the starship palette
- **THEN** the next shell prompt render uses the new colors

### Requirement: Per-invocation tools update on next run
The sync script SHALL update config files for bat, delta, and eza. These tools read config on each invocation and SHALL use new colors on their next run without any reload signal.

#### Scenario: Next eza invocation uses new colors
- **WHEN** the sync script updates the eza theme config
- **THEN** the next `eza` command uses the new colors

#### Scenario: Next delta invocation uses new colors
- **WHEN** the sync script updates the delta git include
- **THEN** the next `git diff` uses the new colors

### Requirement: SwayOSD restart on theme change
The sync script SHALL rewrite the swayosd CSS file and restart the swayosd service. SwayOSD does not support live config reload.

#### Scenario: SwayOSD applies new theme
- **WHEN** the sync script writes the swayosd CSS and restarts the service
- **THEN** the next OSD popup uses the new colors
