# OpenTofu Library Helper Scripts Testing Plan

## Overview

This testing plan validates the abstracted helper scripts generated by the OpenTofu library to ensure they work correctly with the new deployment service architecture. The testing covers functionality, service integration, lock management, error handling, and backward compatibility.

## Current Architecture Analysis

### Service Structure
- **Main Service**: `keycloak.service`
- **Deployment Service**: `keycloak-terraform-deploy-adeci.service` (new blocking deployment pattern)
- **Instance Name**: `adeci`
- **State Directory**: `/var/lib/keycloak-adeci-terraform`

### Helper Scripts (Generated by OpenTofu Library)
1. `keycloak-tf-unlock-adeci` - Force remove terraform state locks
2. `keycloak-tf-status-adeci` - Show lock status and service health
3. `keycloak-tf-apply-adeci` - Manual terraform deployment trigger
4. `keycloak-tf-logs-adeci` - Monitor terraform execution

## Testing Categories

## 1. Functional Testing

### 1.1 Script Availability Testing
**Objective**: Verify all scripts are available in PATH and executable

```bash
# Test 1.1.1: Script existence
which keycloak-tf-unlock-adeci
which keycloak-tf-status-adeci
which keycloak-tf-apply-adeci
which keycloak-tf-logs-adeci

# Test 1.1.2: Script permissions
ls -la $(which keycloak-tf-unlock-adeci)
ls -la $(which keycloak-tf-status-adeci)
ls -la $(which keycloak-tf-apply-adeci)
ls -la $(which keycloak-tf-logs-adeci)

# Test 1.1.3: Script help/basic execution
keycloak-tf-unlock-adeci --help || echo "No help flag - OK"
keycloak-tf-status-adeci  # Should run without errors
keycloak-tf-logs-adeci --help || echo "No help flag - OK"
```

**Expected Results**:
- All scripts exist in PATH
- All scripts are executable
- Status script runs without errors
- Other scripts show appropriate usage or run safely

### 1.2 Status Script Functionality
**Objective**: Verify status script shows correct information

```bash
# Test 1.2.1: Basic status display
keycloak-tf-status-adeci

# Test 1.2.2: Status with no locks (clean state)
systemctl stop keycloak-terraform-deploy-adeci.service
rm -f /var/lib/keycloak-adeci-terraform/.terraform.lock*
keycloak-tf-status-adeci

# Test 1.2.3: Status with active service
systemctl start keycloak.service
keycloak-tf-status-adeci
```

**Expected Results**:
- Shows lock status (active/none)
- Shows deployment service status
- Shows main service status
- Clean formatting and clear information

### 1.3 Apply Script Functionality
**Objective**: Verify apply script triggers deployments correctly

```bash
# Test 1.3.1: Manual deployment trigger (clean state)
systemctl stop keycloak-terraform-deploy-adeci.service
rm -f /var/lib/keycloak-adeci-terraform/.deploy-complete
keycloak-tf-apply-adeci &
APPLY_PID=$!

# Verify service starts
sleep 2
systemctl is-active keycloak-terraform-deploy-adeci.service

# Test 1.3.2: Apply with service already running
keycloak-tf-apply-adeci &
APPLY_PID2=$!
```

**Expected Results**:
- Removes .deploy-complete flag
- Starts deployment service
- Follows logs automatically
- Handles concurrent executions gracefully

### 1.4 Unlock Script Functionality
**Objective**: Verify unlock script handles lock removal correctly

```bash
# Test 1.4.1: Unlock with no locks
keycloak-tf-unlock-adeci

# Test 1.4.2: Create test lock and unlock
mkdir -p /var/lib/keycloak-adeci-terraform
touch /var/lib/keycloak-adeci-terraform/.terraform.lock
echo -e "PID: 12345\nDate: $(date -Iseconds)\nService: test" > /var/lib/keycloak-adeci-terraform/.terraform.lock.info
echo "y" | keycloak-tf-unlock-adeci

# Test 1.4.3: Cancel unlock operation
touch /var/lib/keycloak-adeci-terraform/.terraform.lock
echo "n" | keycloak-tf-unlock-adeci
```

**Expected Results**:
- Shows "No lock files found" when no locks exist
- Displays lock information when locks exist
- Removes locks when confirmed
- Preserves locks when cancelled

### 1.5 Logs Script Functionality
**Objective**: Verify logs script monitors execution correctly

```bash
# Test 1.5.1: Logs with inactive service
systemctl stop keycloak-terraform-deploy-adeci.service
timeout 5 keycloak-tf-logs-adeci || echo "Timeout expected"

# Test 1.5.2: Logs with active service
systemctl start keycloak-terraform-deploy-adeci.service &
timeout 10 keycloak-tf-logs-adeci || echo "Timeout expected"
```

**Expected Results**:
- Shows current status information
- Shows deploy-complete flag status
- Shows lock status
- Follows logs with clear output

# Test 2.1.1: Verify deployment service name
systemctl list-units | grep keycloak-terraform-deploy

# Test 2.1.2: Scripts use correct service name

**Expected Results**:
- Deployment service exists with name `keycloak-terraform-deploy-adeci`
- Scripts reference correct service name

**Objective**: Verify scripts work with service dependencies

```bash
# Test 2.2.1: Main service dependency
systemctl stop keycloak.service

sleep 5
systemctl status keycloak-terraform-deploy-adeci.service

# Test 2.2.2: Service startup order
systemctl start keycloak.service
systemctl start keycloak-terraform-deploy-adeci.service
keycloak-tf-status-adeci
```

**Expected Results**:
- Scripts show service status accurately
- Deployment service waits for main service
- Status reflects service dependencies

### 2.3 State Directory Integration
**Objective**: Verify scripts work with correct state directories

```bash
# Test 2.3.1: State directory structure
ls -la /var/lib/keycloak-adeci-terraform/
keycloak-tf-status-adeci

# Test 2.3.2: State file interaction
touch /var/lib/keycloak-adeci-terraform/.deploy-complete
keycloak-tf-status-adeci
keycloak-tf-apply-adeci &

```

**Expected Results**:
- Scripts access correct state directory
- Scripts interact with state files properly
- State changes are reflected in script output

## 3. Lock Interaction Testing

### 3.1 Lock File Management
**Objective**: Verify scripts handle lock files correctly

```bash
# Test 3.1.1: Lock detection
keycloak-tf-status-adeci  # No locks
touch /var/lib/keycloak-adeci-terraform/.terraform.lock
keycloak-tf-status-adeci  # Lock detected

# Test 3.1.2: Lock info display
cat > /var/lib/keycloak-adeci-terraform/.terraform.lock.info <<EOF
PID: $$
Date: $(date -Iseconds)
Service: keycloak-terraform-deploy-adeci
User: root
EOF
keycloak-tf-status-adeci
```

**Expected Results**:
- Accurately detects lock presence
- Displays lock information correctly
- Shows process status (running/not running)

### 3.2 Concurrent Lock Testing
**Objective**: Verify scripts handle concurrent operations

```bash
# Test 3.2.1: Concurrent deployment attempts
keycloak-tf-apply-adeci &
APPLY_PID1=$!
sleep 2
keycloak-tf-apply-adeci &
APPLY_PID2=$!

# Test 3.2.2: Status during locked operation
sleep 2
keycloak-tf-status-adeci

# Test 3.2.3: Unlock during operation
echo "y" | keycloak-tf-unlock-adeci
```

**Expected Results**:
- Second apply waits or fails gracefully
- Status shows active lock during operation
- Unlock removes locks and stops operations

### 3.3 Stale Lock Detection
**Objective**: Verify scripts detect and handle stale locks

```bash
# Test 3.3.1: Create stale lock
cat > /var/lib/keycloak-adeci-terraform/.terraform.lock.info <<EOF
PID: 99999
Date: $(date -Iseconds)
Service: keycloak-terraform-deploy-adeci
User: root
EOF
touch /var/lib/keycloak-adeci-terraform/.terraform.lock
keycloak-tf-status-adeci

# Test 3.3.2: Cleanup stale lock
echo "y" | keycloak-tf-unlock-adeci
```

**Expected Results**:
- Detects stale lock (non-existent PID)
- Shows warning about stale process
- Allows cleanup of stale locks

## 4. Error Handling Testing

### 4.1 Missing Services
**Objective**: Verify scripts handle missing services gracefully

```bash
# Test 4.1.1: Missing main service
systemctl mask keycloak.service
keycloak-tf-status-adeci
systemctl unmask keycloak.service

# Test 4.1.2: Missing deployment service
systemctl mask keycloak-terraform-deploy-adeci.service
keycloak-tf-apply-adeci
systemctl unmask keycloak-terraform-deploy-adeci.service
```

**Expected Results**:
- Scripts don't crash on missing services
- Show appropriate error messages
- Return non-zero exit codes for errors

### 4.2 Missing Files/Directories
**Objective**: Verify scripts handle missing files gracefully

```bash
# Test 4.2.1: Missing state directory
rm -rf /var/lib/keycloak-adeci-terraform
keycloak-tf-status-adeci
keycloak-tf-unlock-adeci

# Test 4.2.2: Permission issues
mkdir -p /var/lib/keycloak-adeci-terraform
chmod 000 /var/lib/keycloak-adeci-terraform
keycloak-tf-status-adeci
chmod 755 /var/lib/keycloak-adeci-terraform
```

**Expected Results**:
- Scripts handle missing directories
- Show appropriate messages for permission issues
- Don't create directories unexpectedly

### 4.3 Invalid Operations
**Objective**: Verify scripts handle invalid operations

```bash
# Test 4.3.1: Apply when service is masked
systemctl mask keycloak-terraform-deploy-adeci.service
keycloak-tf-apply-adeci
systemctl unmask keycloak-terraform-deploy-adeci.service

# Test 4.3.2: Status with corrupted lock files
echo "invalid content" > /var/lib/keycloak-adeci-terraform/.terraform.lock.info
keycloak-tf-status-adeci
```

**Expected Results**:
- Scripts fail gracefully on invalid operations
- Show clear error messages
- Don't leave system in inconsistent state

## 5. Backward Compatibility Testing

### 5.1 Functionality Preservation
**Objective**: Verify scripts provide same functionality as manual operations

```bash

systemctl status keycloak-terraform-deploy-adeci.service --no-pager -l

# Test 5.1.2: Manual vs script deployment

# Compare with:
keycloak-tf-apply-adeci
```

**Expected Results**:
- Scripts provide equivalent information to manual commands
- Scripts trigger same operations as manual systemctl
- Output format is user-friendly

### 5.2 Legacy Service Name Handling
**Objective**: Verify no references to old service names

```bash
# Test 5.2.1: Check for legacy service references
grep -r "keycloak-terraform-adeci" $(which keycloak-tf-*)
# Should find no matches (legacy name)

# Test 5.2.2: Verify current service name usage
grep -r "keycloak-terraform-deploy-adeci" $(which keycloak-tf-*)
# Should find matches in apply and status scripts
```

**Expected Results**:
- No references to legacy service names
- All scripts use current service naming
- Scripts work with new deployment service

### 5.3 Operational Equivalence
**Objective**: Verify scripts achieve same operational goals

```bash
# Test 5.3.1: Deployment trigger equivalence
# Manual method:
rm -f /var/lib/keycloak-adeci-terraform/.deploy-complete
systemctl start keycloak-terraform-deploy-adeci.service

# Script method:

# Test 5.3.2: Status check equivalence
# Manual method:

# Script method:

```

**Expected Results**:
- Scripts achieve same operational results
- Scripts provide additional convenience features
- Scripts maintain same safety guarantees

## Test Execution Strategy

### Prerequisites
1. Deploy keycloak service to test machine (aspen1)
2. Ensure terraform configuration is present
3. Have necessary permissions for systemctl operations
4. Backup any existing state before testing

### Test Phases
1. **Phase 1**: Basic functionality (Tests 1.1-1.5)
2. **Phase 2**: Service integration (Tests 2.1-2.3)
3. **Phase 3**: Lock management (Tests 3.1-3.3)
4. **Phase 4**: Error handling (Tests 4.1-4.3)
5. **Phase 5**: Compatibility validation (Tests 5.1-5.3)

### Success Criteria
- All helper scripts are accessible and executable
- Scripts show correct service status information
- Scripts handle locks correctly (detection, removal, stale cleanup)
- Scripts trigger terraform deployments successfully
- Error conditions are handled gracefully
- No references to legacy service names
- Operational equivalence with manual methods is maintained

### Test Environment
- **Machine**: aspen1 (configured with keycloak service)
- **Instance**: adeci
- **State Directory**: `/var/lib/keycloak-adeci-terraform`
- **Services**: `keycloak.service`, `keycloak-terraform-deploy-adeci.service`

### Cleanup
After testing, ensure:
- Remove any test lock files
- Reset service states to clean condition
- Clear any temporary state modifications
- Verify services are in expected operational state

## Issues to Validate

Based on the current implementation, specific validation points:

1. **Service Name Fixes**: Confirm scripts reference `keycloak-terraform-deploy-adeci.service`
2. **Lock Management**: Verify scripts work with flock-based state locking
3. **Status Checking**: Ensure status script shows deployment service status
4. **Manual Triggers**: Confirm apply script can trigger deployments via deployment service
5. **Helper Script Integration**: Verify the new `mkHelperScripts` function generates working scripts

This comprehensive testing plan ensures the abstracted helper scripts provide the same operational value as manual implementations while working correctly with the new deployment service architecture.
